# User Identity Model

## How Hydra Knows the User

Hydra has no user store and no concept of who a user is. The Login App tells it. After verifying credentials, the Login
App accepts the Hydra login challenge and explicitly passes the user identifier:

```json
{
  "subject": "uuid-123",
  "remember": true,
  "remember_for": 3600
}
```

Hydra stores this as the session key. On subsequent requests (`prompt=none`, SSO), Hydra finds the session by cookie,
retrieves the `subject`, and skips the Login App entirely. The `subject` becomes the `sub` claim in all issued tokens.

---

## The Subject Must Be Globally Unique

Hydra's session store is global — not tenant-scoped. If two users in different tenants shared the same `subject`, their
sessions would collide. The `subject` must therefore be a **globally unique, stable, platform-wide identifier** — a UUID
generated by IDM-2 at user creation time.

Tenant context is separate. A user belonging to multiple tenants has one UUID but multiple tenant memberships. The
selected tenant is injected as a claim at consent time, not encoded in the subject.

---

## The Problem with Federated Users

For locally created users, IDM-2 generates the UUID at creation time. For federated users (SAML, OIDC), the user exists
in the corporate IdP — IDM-2 may have no record of them until they first log in.

This is solved by **Just-in-Time (JIT) provisioning**: on first federated login, the Login App creates a user record in
IDM-2, assigns a UUID, and links the IdP identity to it. On subsequent logins, the existing record is found and the UUID
is reused.

---

## The Linked Identity Model

Every user in IDM-2 has a UUID (surrogate key) and one or more **linked identities** (natural keys). A linked identity
records where the user came from:

```
(tenant_id, provider, external_id)  →  internal UUID
```

This is the lookup path the Login App uses on every login — federated or local.

### Natural Key

The natural key is the combination of three fields:

| Field         | Meaning                                         |
| ------------- | ----------------------------------------------- |
| `tenant_id`   | The tenant the user belongs to                  |
| `provider`    | The identity system that authenticated the user |
| `external_id` | The identifier the provider uses for this user  |

`(tenant_id, provider, external_id)` is enforced as a unique constraint — no duplicate entries. One user may have
multiple linked identity rows (e.g. local + SAML after a tenant migrates to SSO), but each row is unique.

### Surrogate Key

The UUID is the surrogate key — assigned once at user creation or first JIT login, never changes, never reused. This is
what Hydra receives as `subject` and what all downstream services use to reference the user. It is deliberately
decoupled from any mutable attribute.

---

## Provider Values

The `provider` field identifies the identity system. The convention is `{protocol}:{issuer-or-domain}`.

| Provider value                               | Scenario                                         |
| -------------------------------------------- | ------------------------------------------------ |
| `local:idm`                                  | User created and managed directly in IDM-2       |
| `saml:adfs.acmecorp.com`                     | Corporate ADFS, identified by its SAML entity ID |
| `saml:okta.acmecorp.com`                     | Okta as SAML IdP                                 |
| `saml:azure.acmecorp.com`                    | Azure AD as SAML IdP                             |
| `saml:shibboleth.university.edu`             | Shibboleth IdP (higher education)                |
| `oidc:accounts.google.com`                   | Google as OIDC provider (issuer URL)             |
| `oidc:login.microsoftonline.com/{tenant-id}` | Azure AD as OIDC provider                        |
| `oidc:okta.acmecorp.com`                     | Okta as OIDC provider                            |
| `oidc:auth.pingidentity.com`                 | PingIdentity as OIDC provider                    |
| `scim:okta.acmecorp.com`                     | User pre-provisioned via SCIM push               |

For OIDC providers, use the `iss` claim value — it is already the canonical issuer identifier. For SAML providers, use
the `entityID` from the IdP metadata — same reasoning.

---

## External ID Values

The `external_id` field is the identifier the provider uses for the user.

| Provider type         | Recommended external_id  | Notes                                                                        |
| --------------------- | ------------------------ | ---------------------------------------------------------------------------- |
| `local:idm`           | Email address            | Pragmatic, familiar to users. See considerations below                       |
| SAML (ADFS, Azure AD) | Persistent `NameID`      | `urn:oasis:names:tc:SAML:2.0:nameid-format:persistent` — stable opaque ID    |
| SAML (ADFS)           | `objectGUID`             | Stable AD object identifier sent as custom attribute                         |
| SAML (Okta)           | Okta user ID             | Stable internal Okta ID sent as `NameID` or custom attribute                 |
| SAML (Shibboleth)     | `eduPersonPrincipalName` | `username@institution.edu` — standard in higher education                    |
| OIDC (any)            | `sub` claim              | OIDC spec defines `sub` as the stable unique identifier — always prefer this |
| OIDC (Azure AD)       | `oid` claim              | More stable than `sub` across Azure AD multi-app scenarios                   |
| SCIM                  | SCIM `id`                | The provider's stable user ID sent during provisioning                       |

---

## Recommendations

**Local users (`local:idm`)**: use email address as `external_id`.

- Email is what users know as their login identity
- Stable enough for B2B SaaS — email changes are infrequent and handled by updating the linked identity record
- The UUID surrogate key insulates all downstream references from email changes

**OIDC providers**: always use `sub` (or `oid` for Azure AD).

**SAML providers**: prefer persistent `NameID` or a stable opaque attribute (`objectGUID`, Okta user ID) over email —
email is mutable and not always present in SAML assertions.

---

## Considerations for Email as external_id

**Email change**: update `external_id` in the linked identity record. UUID is unchanged — no downstream impact. Requires
an admin flow.

**Email reuse**: a deprovisioned user's email assigned to a new person. Mitigation: soft-delete/deactivate users, check
account status on JIT lookup before reusing a record.

**Case sensitivity**: normalise to lowercase before storing and comparing — apply consistently everywhere.

**Email not always available in SAML**: some IdPs send only a persistent opaque `NameID`. Email as `external_id` applies
to `local:idm` only. For SAML/OIDC providers, use the provider's own stable identifier.

---

## Example Records

```
-- Local user, belongs to one tenant
tenant_id    provider      external_id           internal_uuid
acmecorp     local:idm     alice@acmecorp.com    uuid-123

-- Federated user, SAML SSO via corporate ADFS
tenant_id    provider                  external_id    internal_uuid
acmecorp     saml:adfs.acmecorp.com   S-1-5-21-...   uuid-456

-- Consultant belonging to two tenants (same UUID, two linked identity rows)
tenant_id    provider      external_id           internal_uuid
acmecorp     local:idm     bob@consulting.com    uuid-789
globex       local:idm     bob@consulting.com    uuid-789

-- User with both local and SAML identity (tenant migrated to SSO)
tenant_id    provider                  external_id           internal_uuid
acmecorp     local:idm                 alice@acmecorp.com    uuid-123
acmecorp     saml:adfs.acmecorp.com   S-1-5-21-...          uuid-123
```

The last example shows one user with two linked identities — both resolve to the same UUID. No duplicate natural keys —
each `(tenant_id, provider, external_id)` combination is unique.
