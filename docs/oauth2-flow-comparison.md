# OAuth 2.0 Flow Comparison: Authorization Code vs Client Credentials

| Step                             | Authorization Code (Human)                                                                                                                | Client Credentials (Machine)                                                                                            |
| -------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------- |
| **1. Registration**              | Developer registers app on Auth Server — gets `client_id`, `client_secret`, registers redirect URIs                                       | Service registers on Auth Server — gets `client_id`, `client_secret`. No redirect URI needed                            |
| **2. Initiate flow**             | App builds authorization URL with `response_type=code`, `scope`, `state`, `redirect_uri` and redirects user's browser to `/auth` endpoint | Service POSTs directly to `/token` endpoint with `grant_type=client_credentials`, `client_id`, `client_secret`, `scope` |
| **3. User login**                | Auth Server shows login screen — user enters credentials                                                                                  | N/A                                                                                                                     |
| **4. Consent**                   | Auth Server shows consent screen — user approves requested scopes                                                                         | N/A                                                                                                                     |
| **5. Authorization code issued** | Auth Server redirects browser back to `redirect_uri` with `?code=ABC&state=xyz`                                                           | N/A                                                                                                                     |
| **6. State validation**          | App verifies `state` matches what it sent in Step 2 (CSRF protection)                                                                     | N/A                                                                                                                     |
| **7. Code exchange**             | App POSTs to `/token` with `grant_type=authorization_code`, `code`, `client_id`, `client_secret`, `redirect_uri`                          | N/A                                                                                                                     |
| **8. Token issued**              | Auth Server validates code + secret + redirect URI → returns `access_token`, `refresh_token`, (+ `id_token` if `openid` in scope)         | Auth Server validates `client_id` + `client_secret` → returns `access_token` only. No refresh token, no ID token        |
| **9. Call Resource API**         | App calls API with `Authorization: Bearer <access_token>`                                                                                 | Service calls API with `Authorization: Bearer <access_token>`                                                           |
| **10. Token expires**            | App POSTs to `/token` with `grant_type=refresh_token` + `refresh_token` to get a new access token silently                                | Service repeats Step 2 — just requests a new token directly                                                             |
